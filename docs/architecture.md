# Architecture

Meshery Server installs the Meshery Operator into a cluster on connection and
thereafter manages its health and configuration. The operator owns two custom
resources and reconciles each into concrete workloads.

```
Meshery Server ──(installs & manages via pkg/client typed clientset)──▶ Meshery Operator (manager Deployment)
                                                                          │
                        ┌─────────────────────────────────────────────────┴───────────────┐
                        ▼                                                                   ▼
                 Broker controller                                                  MeshSync controller
                 reconciles Broker CR                                               reconciles MeshSync CR
                        │                                                                   │
            ┌───────────┼───────────────┐                                                   ▼
            ▼           ▼               ▼                                              Deployment
       StatefulSet   Service        2x ConfigMap                                       meshery/meshsync
        (NATS)       (client)       (nats.conf, accounts)                              BROKER_URL env
                        │
                        └──▶ endpoint derivation ──▶ Broker.Status.Endpoint.{Internal,External}
                                                              │
                                                              └──▶ injected as the MeshSync broker URL
```

## Custom resources (`api/v1alpha1`)

- **`Broker`** (`brokers.meshery.io`) - a NATS message broker. Spec today is just
  `Size`; `Status` carries `Endpoint{Internal,External}` and typed `Conditions`.
- **`MeshSync`** (`meshsyncs.meshery.io`) - the cluster-state synchronizer. Spec
  carries `Size`, `Version`, a `Broker` reference (native or custom URL), and a
  `WatchList`. `Status` carries `PublishingTo` and `Conditions`.

Both are namespaced, `apiextensions.k8s.io/v1`, group `meshery.io`, generated by
`controller-gen`. The CRD bases live in `config/crd/bases/`.

## Controllers (`controllers/`)

Both reconcilers follow the same shape (`Reconcile` → finalizer → main reconcile
→ health check → status patch) and implement:

- **Finalizers** - owned objects are cleaned up on delete before the finalizer
  clears.
- **Typed status `Conditions`** via `meta.SetStatusCondition`.
- **An "update existing owned object" path** - on reconcile, existing owned
  objects are synced toward the desired spec.

`main.go`/`cmd/main.go` wires both reconcilers into a single controller-runtime
manager with leader election (stable lease `meshery-operator-leader.meshery.io`),
`healthz`/`readyz`, and a metrics server.

### Known structural debt (tracked in the modernization plan)

- The reconcile sync overwrites `existing.Spec = desired.Spec` and gates on
  `Semantic.DeepEqual`, which fights server-defaulted fields and can hot-loop.
  Server-Side Apply replaces this in WS-3 (#785).
- Only the primary workload is `Owns()`'d; the broker `Service`/`ConfigMap`s are
  not watched, so endpoint changes are not re-triggered (WS-3).
- Endpoint derivation performs blocking TCP dials in the reconcile path; it
  becomes a pure, non-blocking function over the `Service` in WS-3/WS-4.

## Resource builders (`pkg/broker`, `pkg/meshsync`)

Workload manifests are hand-authored Go structs (no Helm / embedded YAML):

- `pkg/broker/resources.go` - the NATS `StatefulSet`, client `Service`, and the
  `nats.conf`/accounts `ConfigMap`s.
- `pkg/meshsync/resources.go` - the MeshSync `Deployment`.

`pkg/broker/broker.go` derives the broker endpoint and checks health via
`ReadyReplicas`. `pkg/meshsync/meshsync.go` injects the broker URL and checks
health.

## Typed client (`pkg/client/v1alpha1`)

A hand-rolled typed clientset (`Brokers(ns)`, `MeshSyncs(ns)` with
Create/Get/Update/Delete/List/Watch/Patch) exported for **Meshery Server**, which
is the operator's primary consumer. Any change to the `v1alpha1` contract must
preserve this surface; see the modernization plan's compatibility section.

## Packaging

- `config/` - Kustomize bases assembled by `config/default`.
- `bundle/` - the OLM bundle (ClusterServiceManifest + CRDs) for operator-catalog
  distribution, generated by `operator-sdk` from `config/manifests`.
